# Offline Capture Analysis

This folder contains tooling for repeatable, table-oriented analysis of the
large PulseView/sigrok example captures.

The analyzer streams `.sr` files directly, decodes UART/RS485 or Classic CAN
frames from the selected logic channel, reuses the protocol helper modules from
`decoders/`, and writes CSV/Markdown outputs.

## Usage

```powershell
python analysis/analyze_capture.py --list-targets
python analysis/analyze_capture.py --all-captures --quiet
python analysis/analyze_capture.py --comparison-only --update-readme-reports
```

Single-capture analysis can usually infer the protocol from the capture name:

```powershell
python analysis/analyze_capture.py examples/bridge/china-tower-modbus-rs485-raw-capture.sr --quiet
python analysis/analyze_capture.py examples/bridge/growatt-can-raw-capture.sr --quiet
```

Override decode settings when needed:

```powershell
python analysis/analyze_capture.py examples/bridge/china-tower-modbus-rs485-raw-capture.sr --protocol china_tower_modbus --channel CH0 --baud 9600 --invert
python analysis/analyze_capture.py examples/bridge/growatt-can-raw-capture.sr --protocol growatt_can --channel CH0 --bitrate 500000 --sample-point 70 --invert
```

Outputs are written under `analysis/out/` by default:

- `*.frames.csv` has one row per decoded frame.
- `*.sequences.csv` has request/response pairing for UART/RS485 captures, or
  CAN cycle grouping for CAN captures.
- `*.summary.md` has quick counts and latency statistics.
- `bridge-analysis-overview.csv` is generated by `--all-captures` and compares
  all supported captures in one table.
- `topology-comparison.csv` compares matching Bridge, Bridge Forward, and Direct
  cable captures by topology.
- `topology-three-mode-comparison.csv` pivots the complete three-mode groups and
  shows Forward/Direct deltas against Bridge.

Readable generated reports are written under `analysis/reports/`. The batch run
also writes [Capture Analysis Overview](reports/bridge-analysis-overview.md) and
[Topology Comparison Report](reports/topology-comparison.md).

Important UART/RS485 timing fields:

```text
request start ---- request end    response start ---- response end
              <-- request_to_response -->
<---------------------- full_exchange ----------------->
```

- `request_to_response_us`: response start minus request end.
- `full_exchange_us`: response end minus request start.
- `gap_from_previous_us`: current frame start minus previous frame end.

Important CAN timing fields:

- `cycle_duration_us`: first frame start to last frame end inside a CAN burst.
- `inter_cycle_gap_us`: current CAN burst start minus previous CAN burst end.
- `gap_from_previous_us`: current frame start minus previous frame end.

Statistics columns:

- `n`: number of complete request/response sequences included.
- `avg`: arithmetic mean across those sequences.
- `min`: fastest observed value.
- `max`: slowest observed value.
- `p50`: median value; half of the observations are below this value.
- `p95`: 95th percentile; 95% of observations are below this value.

## Design Notes

The scanner does not loop over every sample in Python. It translates the selected
probe byte to a 0/1 level buffer and finds transitions in C-backed operations,
then the UART decoder only samples scheduled bit centers. That keeps large
captures practical while preserving sample-accurate timing.

For CAN, the analyzer uses the same transition stream, samples each bit at a
configurable sample point, destuffs Classic CAN frames, validates CRC, and groups
bursts with `--cycle-gap-us`.
